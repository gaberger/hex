# Approach A: Embedded Infisical Packaging

> hex ships a Docker Compose stack that runs Infisical + Postgres + Redis alongside hex's dashboard. A single `hex serve` command starts everything.

## 1. Overview

In this approach, hex embeds Infisical as a first-class managed service. When a user runs `hex setup --secrets`, hex pulls the required Docker images and generates a local configuration. From that point forward, `hex serve` starts the dashboard hub (port 5555) **and** the Infisical stack (Infisical on port 8080, Postgres on 5433, Redis on 6380). Agents resolve secrets via `ISecretsPort` backed by the `InfisicalAdapter`, which points at `http://localhost:8080` by default.

This is the "batteries-included" option: zero external accounts, works offline, single command to start.

## 2. Installation Flow

### 2.1. `hex setup --secrets`

```bash
hex setup --secrets
```

**Steps executed:**

1. **Docker check** -- Verify `docker` and `docker compose` are available. If missing, print a clear error with install instructions and exit 1.

2. **Pull images** -- Pull pinned image versions (not `latest`) to ensure reproducibility:
   ```
   infisical/infisical:v0.86.0
   postgres:15-alpine
   redis:7-alpine
   ```

3. **Generate config** -- Create `.hex/secrets-config.json` (gitignored) with:
   ```json
   {
     "version": 1,
     "infisical": {
       "siteUrl": "http://localhost:8080",
       "postgresPort": 5433,
       "redisPort": 6380,
       "encryptionKey": "<generated-32-byte-hex>",
       "authSecret": "<generated-64-byte-hex>",
       "postgresPassword": "<generated-24-byte-hex>"
     },
     "machineIdentity": null
   }
   ```

4. **First-run wizard** -- On first `hex serve` after setup (when `machineIdentity` is null):
   - Start the Docker stack
   - Wait for Infisical health endpoint
   - Use the Infisical admin setup API (`POST /api/v1/signup`) to create the initial admin account with a generated password
   - Create a machine identity via API (`POST /api/v2/identities`)
   - Create a universal auth credential for the machine identity
   - Create a default project ("hex-secrets")
   - Grant the machine identity access to the project
   - Store the resulting `clientId`, `clientSecret`, and `projectId` in `.hex/secrets-config.json` under `machineIdentity`
   - Print the admin password once (user must save it)

### 2.2. Gitignore Safety

`hex setup --secrets` appends to `.gitignore`:
```
# hex secrets (embedded Infisical)
.hex/secrets-config.json
.hex/secrets-data/
```

## 3. Docker Compose Configuration

File: `.hex/docker-compose.secrets.yml` (generated by `hex setup --secrets`)

```yaml

# Managed by hex — do not edit manually
# Generated by: hex setup --secrets

name: hex-secrets

services:
  infisical:
    image: infisical/infisical:v0.86.0
    container_name: hex-infisical
    restart: unless-stopped
    ports:
      - "8080:8080"
    environment:
      - ENCRYPTION_KEY=${HEX_INFISICAL_ENCRYPTION_KEY}
      - AUTH_SECRET=${HEX_INFISICAL_AUTH_SECRET}
      - DB_CONNECTION_URI=postgresql://infisical:${HEX_POSTGRES_PASSWORD}@hex-postgres:5432/infisical
      - REDIS_URL=redis://hex-redis:6379
      - SITE_URL=http://localhost:8080
      - NODE_ENV=production
      - TELEMETRY_ENABLED=false
    depends_on:
      postgres:
        condition: service_healthy
      redis:
        condition: service_healthy
    healthcheck:
      test: ["CMD", "wget", "--quiet", "--tries=1", "--spider", "http://localhost:8080/api/status"]
      interval: 10s
      timeout: 5s
      retries: 5
      start_period: 30s
    networks:
      - hex-internal
    deploy:
      resources:
        limits:
          cpus: "1.0"

          cpus: "0.25"
          memory: 256M

  postgres:
    image: postgres:15-alpine
    container_name: hex-postgres

    ports:
      - "5433:5432"
    environment:
      - POSTGRES_USER=infisical
      - POSTGRES_PASSWORD=${HEX_POSTGRES_PASSWORD}

    volumes:
      - hex-pg-data:/var/lib/postgresql/data

      test: ["CMD-SHELL", "pg_isready -U infisical"]
      interval: 5s

      - hex-internal

      resources:
        limits:
          cpus: "0.5"

          cpus: "0.1"

    container_name: hex-redis
    restart: unless-stopped
    ports:
      - "6380:6379"
    command: redis-server --maxmemory 64mb --maxmemory-policy allkeys-lru
    volumes:
      - hex-redis-data:/data
    healthcheck:
      test: ["CMD", "redis-cli", "ping"]
      interval: 5s
      timeout: 3s
      retries: 5
    networks:
      - hex-internal
    deploy:
      resources:
        limits:
          cpus: "0.25"
          memory: 96M
        reservations:
          cpus: "0.05"
          memory: 32M

volumes:
  hex-pg-data:
    driver: local
    driver_opts:
      type: none
      o: bind
      device: ${HEX_DATA_DIR}/postgres
  hex-redis-data:
    driver: local
    driver_opts:
```

### Environment Variable Injection

hex reads `.hex/secrets-config.json` and passes values as environment variables to `docker compose` via `--env-file` or programmatic `spawn`:

```typescript
// In DockerSecretsAdapter (new secondary adapter)
async startStack(): Promise<void> {
  const config = await this.readConfig();
  const env = {
    HEX_INFISICAL_ENCRYPTION_KEY: config.infisical.encryptionKey,
    HEX_INFISICAL_AUTH_SECRET: config.infisical.authSecret,
    HEX_POSTGRES_PASSWORD: config.infisical.postgresPassword,
    HEX_DATA_DIR: path.join(this.projectPath, '.hex', 'secrets-data'),
  };

  await execFile('docker', [
    'compose', '-f', this.composePath, 'up', '-d',
  ], { env: { ...process.env, ...env } });
}
```

## 4. CLI Integration

### 4.1. `hex serve` (enhanced)

The existing `hex serve` starts the dashboard hub. With `--secrets` configured, it additionally starts the Docker stack:

```
hex serve
  ├── Start Docker Compose stack (.hex/docker-compose.secrets.yml)
  ├── Wait for Infisical health (up to 60s with exponential backoff)
  ├── Run first-run wizard if machineIdentity is null
  ├── Start Dashboard Hub (:5555)
  └── Register project with dashboard
```

If `--no-secrets` is passed, skip the Docker stack (useful for CI where secrets come from the environment).

### 4.2. `hex secrets` Subcommands

| Command | Description |
|---------|-------------|
| `hex secrets add KEY=VALUE` | Store a secret in the local Infisical instance |
| `hex secrets add KEY=VALUE --env staging` | Store for a specific environment |
| `hex secrets rm KEY` | Delete a secret |
| `hex secrets list` | List keys with metadata (version, updated date). Never shows values. |
| `hex secrets get KEY` | Retrieve a single secret value (printed to stdout for piping) |
| `hex secrets env` | Export all secrets as `KEY=VALUE` lines (for `eval $(hex secrets env)`) |
| `hex secrets env --json` | Export as JSON object |
| `hex secrets status` | Show Infisical stack health, container status, secret count |
| `hex secrets stop` | Stop the Docker stack (data persisted in volumes) |
| `hex secrets destroy` | Stop stack, remove volumes, delete config. Requires `--confirm`. |

### 4.3. Agent Secret Access at Runtime

Agents receive secrets through the `AppContext.secrets` field (already wired via `ISecretsPort`). The existing `InfisicalAdapter` handles this -- no new adapter is needed for reads.

When hex spawns an agent (via `ISwarmPort.spawnAgent`), the composition root injects the same `ISecretsPort` instance. Agents call:

```typescript
const dbUrl = await ctx.secrets.resolveSecret('DATABASE_URL');
if (dbUrl.ok) {
  // use dbUrl.value
}
```

For subprocess-based agents (e.g., running in a worktree), hex exports secrets as environment variables:

```typescript
// In WorkplanExecutor, before spawning a build step
const secrets = await ctx.secrets.listSecrets();
const env: Record<string, string> = {};
for (const meta of secrets) {
  const result = await ctx.secrets.resolveSecret(meta.key);
  if (result.ok) env[meta.key] = result.value;
}
// Pass env to child process
await execFile('bun', ['test'], { env: { ...process.env, ...env } });
```

## 5. Composition Root Changes

### 5.1. Auto-Discovery

`createAppContext()` gains a new detection path **before** the existing env-var check:

```typescript
// Priority order for secrets backend:
// 1. Embedded Infisical (local Docker stack via .hex/secrets-config.json)
// 2. External Infisical (INFISICAL_CLIENT_ID + INFISICAL_CLIENT_SECRET env vars)
// 3. EnvSecretsAdapter fallback

let secrets: ISecretsPort;

// 1. Check for embedded Infisical config
const embeddedConfigPath = `${outputDir}/secrets-config.json`;
const embeddedConfigExists = await fs.exists(
  embeddedConfigPath.replace(projectPath + '/', '')
);

if (embeddedConfigExists) {
  const raw = await fs.read(embeddedConfigPath.replace(projectPath + '/', ''));
  const config = JSON.parse(raw);

  if (config.machineIdentity) {
    // Verify Docker stack is running before connecting
    const healthy = await checkInfisicalHealth(config.infisical.siteUrl);
    if (healthy) {
      secrets = new InfisicalAdapter({
        siteUrl: config.infisical.siteUrl,
        clientId: config.machineIdentity.clientId,
        clientSecret: config.machineIdentity.clientSecret,
        projectId: config.machineIdentity.projectId,
        defaultEnvironment: 'dev',
      });
      process.stderr.write(
        `[hex] Secrets: embedded Infisical (${config.infisical.siteUrl})\n`
      );
    } else {
      process.stderr.write(
        `[hex] WARNING: Embedded Infisical configured but not running. ` +
        `Run 'hex serve' to start. Falling back to env vars.\n`
      );
      secrets = new EnvSecretsAdapter();
    }
  } else {
    process.stderr.write(
      `[hex] Secrets config found but not initialized. ` +
      `Run 'hex serve' to complete setup.\n`
    );
    secrets = new EnvSecretsAdapter();
  }
} else if (infisicalClientId && infisicalClientSecret && infisicalProjectId) {
  // 2. External Infisical (existing path — unchanged)
  secrets = new InfisicalAdapter({ /* ... existing config ... */ });
} else {
  // 3. Fallback
  secrets = new EnvSecretsAdapter();
}
```

### 5.2. Health Check Utility

```typescript
async function checkInfisicalHealth(
  siteUrl: string,
  timeoutMs = 3000
): Promise<boolean> {
  try {
    const controller = new AbortController();
    const timer = setTimeout(() => controller.abort(), timeoutMs);
    const res = await fetch(`${siteUrl}/api/status`, {
      signal: controller.signal,
    });
    clearTimeout(timer);
    return res.ok;
  } catch {
    return false;
  }
}
```

### 5.3. Startup Sequence When Using `hex serve`

```
1. Parse .hex/secrets-config.json
2. docker compose -f .hex/docker-compose.secrets.yml up -d
3. Poll GET /api/status every 2s, up to 30 attempts (60s max)
4. If first run: execute first-run wizard (Section 2.1 step 4)
5. Authenticate machine identity → cache access token in memory
6. createAppContext() picks up the embedded config (Section 5.1)
7. Start dashboard hub on :5555
8. Ready.
```

## 6. Upgrade Story

### 6.1. Image Version Pinning

The Docker Compose file uses pinned versions (e.g., `infisical/infisical:v0.86.0`). hex tracks the target version in its own `package.json` under a `hex.infisicalVersion` field.

### 6.2. Upgrade Command

```bash
hex secrets upgrade
```

**Steps:**

1. Read the current pinned version from `.hex/docker-compose.secrets.yml`
2. Read the target version from hex's `package.json` (updated when hex itself is upgraded via npm)
3. If versions differ:
   a. Pull the new image: `docker pull infisical/infisical:v0.87.0`
   b. Stop the stack: `docker compose down`
   c. Update the image tag in `.hex/docker-compose.secrets.yml`
   d. Start the stack: `docker compose up -d`
   e. Infisical runs its own database migrations on startup automatically
4. Print the old and new versions

### 6.3. Data Migration

Infisical handles its own Postgres schema migrations on startup. No hex intervention is needed for minor version bumps. For major version upgrades:

- hex checks Infisical's release notes API for breaking changes
- If a breaking migration is detected, hex prompts the user to back up first:
  ```
  hex secrets backup > infisical-backup-2026-03-16.sql
  ```
- The backup command runs `pg_dump` inside the Postgres container

### 6.4. Rollback

```bash
hex secrets upgrade --rollback
```

Reverts the image tag to the previous version stored in `.hex/secrets-config.json` under `previousVersion`.

## 7. Tradeoffs Analysis

### Pros

| Benefit | Detail |
|---------|--------|
| Zero external setup | No Infisical Cloud account needed. No manual Docker Compose authoring. |
| Works offline | All secrets are local. No network dependency after image pull. |
| Single command | `hex serve` starts everything. Agents get secrets automatically. |
| Consistent environment | Every developer on the team gets the same Infisical version. |
| Production parity | Same Infisical API as production. Secrets can be migrated to hosted Infisical later. |
| Audit trail | Infisical provides secret versioning and access logs out of the box. |

### Cons

| Drawback | Detail |
|----------|--------|
| Docker dependency | Requires Docker Desktop or Docker Engine. Not available on all dev machines. |
| Resource overhead | ~500MB RAM, ~1GB disk for images + data. Non-trivial on 8GB machines. |
| Startup latency | First `hex serve` takes 30-60s for image pull + init. Subsequent starts: 5-15s. |
| Complexity | Three containers to manage. More failure modes than env vars. |
| macOS Docker networking | Docker Desktop on macOS has known DNS and port-binding edge cases. |
| Upgrade coordination | Team members must upgrade hex (and thus Infisical images) in sync. |

### Who This Is Best For

- **Teams** sharing a project where secrets must be consistent across developers
- **CI/CD** pipelines that need a real secrets manager without external dependencies
- **Production-like local dev** where you want the same API surface as your hosted Infisical
- **Air-gapped environments** where cloud secrets managers are not available

### Who Should Use Approach B Instead

- Solo developers who just need a few API keys
- Projects where Docker is not permitted or available
- Environments where 500MB RAM overhead is unacceptable

## 9. Security Considerations

### 9.1. Encryption at Rest

- **Postgres**: Infisical encrypts secret values with AES-256-GCM before storing them. The encryption key is in `.hex/secrets-config.json` (never committed).
- **Redis**: Used only as a cache/queue. No secret values are stored in Redis at rest.
- **Volume mounts**: Bound to `.hex/secrets-data/` inside the project directory, making data locality explicit.

### 9.2. Network Isolation

- All three containers communicate over the `hex-internal` bridge network.
- Postgres and Redis ports are exposed to the host (5433, 6380) for debugging but are **not** exposed outside the host.
- Infisical port 8080 is host-only. In production, this would sit behind a reverse proxy with TLS -- but for local dev, plaintext localhost is acceptable.

### 9.3. Machine Identity Lifecycle

| Event | Action |
|-------|--------|
| `hex setup --secrets` | Machine identity created with scoped project access |
| `hex secrets destroy` | Machine identity revoked, credentials deleted |
| Token expiry | `InfisicalAdapter` already handles token refresh (existing code) |
| Key rotation | `hex secrets rotate-identity` regenerates client credentials |

### 9.4. Credential Storage

All credentials live in `.hex/secrets-config.json`:

- **Encryption key**: AES-256 key for Infisical's server-side encryption
- **Auth secret**: JWT signing key for Infisical sessions
- **Postgres password**: Database credential
- **Machine identity credentials**: clientId + clientSecret for API access

This file is:
- Created with `0600` permissions (owner read/write only)
- Added to `.gitignore` automatically
- Never read by any adapter except the Docker secrets adapter and composition root

### 9.5. Docker Compromise Scenarios

| Scenario | Impact | Mitigation |
|----------|--------|------------|
| Container escape | Attacker has host access | Containers run as non-root; resource limits prevent fork bombs |
| Volume access | Attacker reads Postgres data | Secrets are encrypted at rest with key outside the volume |
| Network sniffing | Attacker reads API traffic | Traffic is localhost-only; no secrets traverse external networks |
| Image supply chain | Malicious image | Pinned versions + digest verification (`docker pull --platform`) |
| `.hex/` directory exposed | All credentials leaked | File permissions (0600) + `.gitignore` + docs warning |

### 9.6. Recommendations

1. **Never run `hex serve` on a shared machine** without restricting port access
2. **Back up `.hex/secrets-config.json`** separately -- losing it means losing the encryption key
3. **Use `hex secrets destroy --confirm`** before decommissioning a project to wipe credentials
4. For production secrets, migrate to Infisical Cloud or a self-hosted instance with TLS -- the embedded stack is for **development only**

## 10. New Adapter: `DockerSecretsAdapter`

This approach requires one new secondary adapter that manages the Docker Compose lifecycle:

```typescript
// src/adapters/secondary/docker-secrets-adapter.ts

export interface IDockerSecretsPort {
  /** Start the embedded Infisical Docker stack */
  startStack(): Promise<void>;

  /** Stop the stack (preserves data) */
  stopStack(): Promise<void>;

  /** Destroy stack + volumes + config */
  destroyStack(confirm: boolean): Promise<void>;

  /** Check if the stack is running and healthy */
  isHealthy(): Promise<boolean>;

  /** Run first-time setup wizard */
  initialize(): Promise<{ clientId: string; clientSecret: string; projectId: string }>;

  /** Get stack status */
  status(): Promise<DockerStackStatus>;
}

interface DockerStackStatus {
  running: boolean;
  containers: { name: string; status: string; health: string }[];
  infisicalVersion: string;
  secretCount: number;
  uptimeSeconds: number;
}
```

This port is consumed by the CLI adapter (for `hex serve` and `hex secrets` commands) and by the composition root (for health checks during startup). It follows hex's architectural rules: the port definition lives in `src/core/ports/`, the implementation in `src/adapters/secondary/`.
