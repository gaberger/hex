# ADR-2026-04-24-1815: Explicit Schema Contract Validation for Daemon→Nexus API Calls

**Status:** Accepted (resolved 2026-05-08)
**Date:** 2026-04-24
**Drivers:** Silent API failure in brain daemon — events POSTed with wrong schema (`"type"` instead of `"event_type"`, missing required `"session_id"`) for ~18 hours before detection. No error surfaced because the daemon used `_ = fire_and_forget()`.

## Context

The brain daemon (`hex sched daemon`) runs as a background process detached from any terminal. It periodically POSTs `brain_tick` events to the nexus REST API (`/api/events`). Two schema violations caused these events to be silently dropped:

1. **Wrong field name**: Daemon sent `"type": "brain_tick"` instead of `"event_type": "brain_tick"` — the `InsertEventRequest` struct in `hex-nexus/src/ports/events.rs` defines the required field as `event_type`.
2. **Missing required field**: `InsertEventRequest.session_id` is required by the schema. The daemon sent no `session_id`, making every POST technically invalid.

The daemon used `_ = client.post(...).send().await` (fire-and-forget) with a 2-second timeout. On schema rejection, the server returns a 400 but the daemon ignores the response. Events were written to a log that nobody watches.

This is a systemic pattern: if any future daemon-to-nexus POST uses the wrong field names or omits required fields, it will silently fail in the exact same way.

## Decision

All daemon-to-nexus API calls MUST be validated against the target schema before deployment. The validation is enforced by two mechanisms:

### 1. Compile-time schema validation (opt-in per call site)

For each POST body, derive from the target `InsertEventRequest` (or equivalent) struct using `serde::Serialize`. This causes a compile error if a field is renamed or a required field is removed.

**Before (fragile):**
```rust
let _ = reqwest::Client::new()
    .post(&event_url)
    .json(&serde_json::json!({
        "type": "brain_tick",  // ← wrong field name, no compiler warning
    }))
    ...
```

**After (schema-validated):**
```rust
let body = serde_json::json!({
    "session_id": session_id,  // required by InsertEventRequest
    "event_type": "brain_tick",  // matches InsertEventRequest::event_type
    "duration_ms": elapsed.as_millis() as i64,
});
let _ = reqwest::Client::new()
    .post(&event_url)
    .json(&body)  // serde_json::Value is Serialize — serde validates at runtime
    ...
```

### 2. Explicit session_id injection

Every daemon that POSTs to `/api/events` MUST populate `session_id`. When `CLAUDE_SESSION_ID` is unset (standalone mode), use a stable identifier derived from the daemon type and PID:

```rust
let session_id = std::env::var("CLAUDE_SESSION_ID")
    .unwrap_or_else(|_| format!("sched-daemon-{}", std::process::id()));
```

### 3. Fire-and-forget replaced with logged failure

Replace `_ = future` with explicit error handling that writes to stderr when the POST fails:

```rust
if let Err(e) = client.post(&event_url).json(&body).timeout(Duration::from_secs(2)).send().await {
    eprintln!("  {} failed to emit brain_tick event: {}", "✗".red(), e);
}
```

This ensures background daemon failures are visible in the daemon log (`~/.hex/logs/`).

## Consequences

**Positive:**
- Schema mismatches caught at build time (if using struct derivation) or at first daemon run (if using `serde_json::Value`)
- Required field omissions cause a visible error instead of silent drops
- Daemon log captures API failures for debugging

**Negative:**
- Slightly more verbose code at each POST site
- Requires developers to know the target schema before writing the POST

**Mitigations:**
- A shared helper function `emit_event(session_id, event_type, payload)` in `sched.rs` centralizes the pattern, reducing repetition
- The `InsertEventRequest` struct in `hex-nexus` is the single source of truth — any rename breaks the compiler on all callers
