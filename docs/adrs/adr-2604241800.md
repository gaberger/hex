# ADR-2604241800: Periodic Dead Code Analysis via Brain Queue

## Status

Proposed

## Context

`hex analyze` catches boundary violations and dead exports, but it's only run manually or during CI. The codebase drifts out of compliance between runs. We need autonomous, scheduled architecture health checks that:
- Run on a configurable interval without human intervention
- Store results in SpacetimeDB for trend tracking
- Alert on regressions (new violations since last run)
- Don't block the daemon tick loop

## Decision

Add an `"analyze"` task kind to the brain queue. The sched daemon tick loop will enqueue one if none is pending and the interval has elapsed. `execute_brain_task` will run `hex analyze . --json` and store the structured result in SpacetimeDB.

### New task kind: `"analyze"`

```rust
// In execute_brain_task:
"analyze" => {
    // Run hex analyze . --json and capture structured output
    std::process::Command::new("hex")
        .args(["analyze", ".", "--json"])
        .output()
}
```

### Daemon tick modification

On each tick:
1. Check if an `analyze` task with status `pending`/`in_progress` already exists for this project — if so, skip.
2. Check if `last_analyze_at` + `ANALYZE_INTERVAL_SECS` has elapsed.
3. If elapsed and no active task, enqueue a new `analyze` task.

### Result storage

After execution, `update_brain_task` stores the full JSON result. A new `GET /api/brain/analysis/history` endpoint serves the last N analysis snapshots for trend display.

### Configuration

```json
// .hex/project.json
{
  "brain": {
    "analyze_interval_secs": 3600,  // default: 1 hour
    "analyze_enabled": true
  }
}
```

## Consequences

### Positive
- Architecture health is continuously monitored, not just on manual invocation
- Regressions are detected and notified automatically
- Results persist for trend analysis
- Zero new infrastructure — reuses existing brain queue, daemon, and SpacetimeDB

### Negative
- `hex analyze` runs periodically even when code isn't changing
- Staggered with existing `validate()` call — two parallel heavy operations per tick when both are enabled

## Alternatives Considered

| Alternative | Why rejected |
|-------------|--------------|
| Separate cron job | Duplicates daemon infrastructure; no coordination with brain queue |
| hex-nexus background service | More complex; hex-nexus is already running the daemon via hex-cli |
| Push-based via git hooks | Only detects violations on commit, not drift between commits |
