# ADR-055: README-Driven Project Specification

**Status:** Accepted
**Accepted Date:** 2026-03-22
## Date: 2026-03-22

- **Informed by**: ADR-043 (project manifest), ADR-008 (dogfooding), ADR-052 (AIIDE)
- **Authors**: Gary (architect), Claude (analysis)

## Context

When a developer runs `hex init` in an empty directory today, hex creates `.hex/project.yaml` (ADR-043) and scaffolds configuration files. But there is no step that captures **what the project is, why it exists, and what it intends to become**. The developer must manually create a README after the fact, and it quickly drifts from reality as ADRs are accepted, features ship, and architecture evolves.

### Problems

1. **No project interview**: `hex init` treats all projects the same — it doesn't ask about scope, language, domain, or constraints. The AI agents that later work on the project have no specification to ground their decisions.

2. **README is an afterthought**: Most README files are written once and never updated. They describe setup steps but not intent, scope, or architectural decisions.

3. **ADRs are invisible**: hex tracks 50+ ADRs, but their impact on the project's direction is scattered across individual files. No single document synthesizes "where we are now" as decisions accumulate.

4. **Agents lack specification context**: When a hex agent starts work, it reads CLAUDE.md for tooling rules but has no project-level specification. It doesn't know the project's goals, constraints, or language philosophy.

### Observation

The README is the one file every developer, every CI system, and every AI agent reads first. If it were treated as the **living specification** — generated from an interview and kept current as ADRs land — it would solve all four problems.

## Decision

### 1. Interactive Project Interview on `hex init` (Empty Directory)

When `hex init` detects an empty or git-init-only directory (no source files), it enters **interview mode** before scaffolding:

```
hex init

⬡ hex — New Project Setup
────────────────────────────────────

What is this project called?
> fleet-manager

Describe the project in 1-2 sentences:
> A distributed fleet management service that coordinates vehicle
  telemetry ingestion, route optimization, and driver dispatch.

What is the primary language?
  [1] Rust
  [2] Go
  [3] TypeScript
  [4] Python
  [5] Multi-language (specify)
> 1

What kind of project is this?
  [1] CLI tool
  [2] Web service / API
  [3] Library / SDK
  [4] Full-stack application
  [5] Infrastructure / DevOps
  [6] Other (describe)
> 2

What are the key constraints or non-negotiables?
> Must handle 10K concurrent WebSocket connections. All state in
  SpacetimeDB. No ORM — direct SQL only. Must run on ARM64.

Any known external dependencies or integrations?
> SpacetimeDB, Redis for caching, Prometheus metrics, S3 for
  telemetry archival.
```

The interview is conversational when running inside Claude Code (the AI agent conducts it), or prompt-based when running from the terminal directly.

### 2. README.md as Living Specification

The interview generates a structured `README.md` that serves as the project specification:

```markdown
# fleet-manager

A distributed fleet management service that coordinates vehicle telemetry
ingestion, route optimization, and driver dispatch.

## Project Scope

- **Type**: Web service / API
- **Primary Language**: Rust
- **Created**: 2026-03-22
- **Status**: Inception

## Architecture

Hexagonal architecture (Ports & Adapters) enforced by hex.

### Constraints
- Must handle 10K concurrent WebSocket connections
- All state in SpacetimeDB — no ORM, direct SQL only
- Must run on ARM64

### External Dependencies
| Dependency | Purpose |
|-----------|---------|
| SpacetimeDB | Primary state store |
| Redis | Caching layer |
| Prometheus | Metrics collection |
| S3 | Telemetry archival |

## Architecture Decisions

<!-- hex:adr-summary — auto-updated by hex -->
No ADRs recorded yet. Use `hex adr create` to record decisions.
<!-- /hex:adr-summary -->

## Getting Started

\`\`\`bash
hex nexus start        # Start the development nexus
hex analyze .          # Check architecture health
cargo run              # Run the service
\`\`\`

## Development

\`\`\`bash
hex validate           # Run semantic validation
cargo test             # Run test suite
\`\`\`

---
*This README is the project specification. It is generated by `hex init`
and kept current as architecture decisions are made. Last updated: 2026-03-22.*
```

### 3. Automatic README Updates on ADR Lifecycle Events

When an ADR transitions to `Accepted`, hex updates the `<!-- hex:adr-summary -->` section in README.md:

```markdown
## Architecture Decisions

<!-- hex:adr-summary — auto-updated by hex -->
| ADR | Title | Status | Date |
|-----|-------|--------|------|
| 001 | Use SpacetimeDB for all state | Accepted | 2026-03-22 |
| 002 | WebSocket-first client protocol | Accepted | 2026-03-23 |
| 003 | ARM64 cross-compilation via cross-rs | Proposed | 2026-03-25 |
<!-- /hex:adr-summary -->
```

This is implemented as a **post-hook** on `hex adr create` and `hex adr status <id> --set accepted`:

```toml
# .hex/hooks.toml (or settings.json hook)
[hooks.post_adr_accept]
command = "hex readme sync-adrs"
```

The hook:
1. Reads all ADR files from `docs/adrs/`
2. Parses status and title from each
3. Regenerates the table between the sentinel comments
4. Writes README.md (only the ADR section — other sections untouched)

### 4. `hex readme` Subcommand

New CLI subcommand for README management:

```bash
hex readme sync-adrs       # Regenerate ADR summary table in README
hex readme sync             # Full sync — ADRs + project status
hex readme interview        # Re-run the project interview (update scope)
```

### 5. Relationship to `.hex/project.yaml` (ADR-043)

The two files serve different audiences:

| File | Audience | Content | Format |
|------|----------|---------|--------|
| `README.md` | Humans + AI agents | Specification, intent, decisions, getting started | Prose Markdown |
| `.hex/project.yaml` | hex tooling | Machine config: name, agent settings, inference endpoints | Structured YAML |

`hex init` generates **both**. The interview feeds into both files:
- Project name, description → both files
- Language choice, constraints → README only
- Agent config, inference endpoints → `.hex/project.yaml` only

### 6. Existing Projects (`hex init` in Non-Empty Directory)

When `hex init` runs in a directory with existing source code:
1. Skip the interview (project already exists)
2. Generate `.hex/project.yaml` as today (ADR-043)
3. If no `README.md` exists, offer to generate one by analyzing the codebase
4. If `README.md` exists, offer to add the `<!-- hex:adr-summary -->` sentinel for ADR tracking

## Implementation Plan

| Phase | Component | Description |
|-------|-----------|-------------|
| 1 | `hex init` interview | Add interactive prompts when directory is empty |
| 2 | README generation | Template engine for structured README from interview answers |
| 3 | `hex readme sync-adrs` | ADR parser + README section updater |
| 4 | Post-ADR hook | Auto-trigger sync on ADR status changes |
| 5 | Agent context | Agents read README.md for project specification on startup |

## Consequences

### Positive
- Every hex project has a specification from day one
- AI agents have project context grounded in intent, not just code structure
- ADR decisions are visible in the one file everyone reads
- README stays current automatically — no manual maintenance
- Interview captures constraints that prevent agents from making wrong assumptions

### Negative
- Interview adds friction to `hex init` for developers who want to skip ahead
- Auto-updating README could conflict with manual edits outside sentinel comments
- README becomes a merge conflict surface if multiple branches modify scope

### Mitigations
- `hex init --skip-interview` flag for developers who want bare scaffolding
- Sentinel comments (`<!-- hex:adr-summary -->`) clearly delineate auto-managed sections
- Only the ADR table is auto-updated; prose sections require explicit `hex readme interview`
