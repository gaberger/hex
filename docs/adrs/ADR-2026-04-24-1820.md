# ADR-2026-04-24-1820: Daemon Staleness Detection — Know When You're Running Stale Code

**Status:** Accepted
**Implementation-Present:** 2026-05-12 by auto-scan — evidence: hex-cli/src/commands/sched.rs
**Date:** 2026-04-24
**Drivers:** Autonomous development rebuilds `hex-cli` or `hex-nexus` without restarting the background daemon. The daemon continues running the old binary, producing stale or wrong results. The operator tests conditions that no longer match the code under test. This pattern has caused silent test failures across multiple sessions (e.g. brain daemon running old binary at 11:49 while new code built at 13:59).

## Context

hex runs several long-lived background processes:
- `hex sched daemon` — brain queue processor, tick loop
- `hex inference watch --daemon` — inference model health monitor
- `hex-nexus` — REST API + WebSocket hub

These are started via `hex sched daemon --background` or similar and detach from the terminal. When an agent rebuilds `hex-cli --release`, the running daemon process is **not** restarted — it keeps the old in-memory code.

This creates a silent mismatch:
- The operator sees new code in source files
- The running daemon executes old code
- Any `hex sched watch`, API queries, or queue behavior reflects the old binary
- Autonomous agents test against stale conditions without knowing it

The current situation: the brain daemon ran for ~18 hours with a schema mismatch (`"type"` vs `"event_type"`) because nobody could tell it was stale. No warning was emitted.

## Decision

Every background daemon MUST be able to detect when its own binary has been rebuilt and surface this explicitly. Two layers:

### Layer 1 — Daemon self-check (runs each tick)

On every tick, the daemon compares its binary's modification time against a persistent record of the mtime it had at startup. A mismatch means the binary was rebuilt while the daemon was running — a staleness signal.

```
~/.hex/daemon-staleness.json:
{
  "sched-daemon": {
    "binary": "/var/home/gary/hex-intf/target/release/hex",
    "startup_mtime": "2026-04-24T14:41:00Z",
    "startup_pid": 3597004
  }
}
```

If `current_mtime != startup_mtime`, log a warning on next tick:
```
⬡ WARNING: daemon binary was rebuilt while running (pid=3597004).
  Restart with: hex sched daemon restart
  Skipping this tick to prevent stale execution.
```

The daemon then skips execution of the current tick (does NOT run validate, drain queue, or emit events) until restarted. This prevents a stale daemon from producing wrong results.

### Layer 2 — CLI staleness check (runs on each command)

Before any command that reads daemon state (`hex sched watch`, `hex sched queue list`, `hex brain daemon-status`, etc.), the CLI checks if the daemon binary has been rebuilt since the daemon started. If so, emit a warning:

```
⚠ Daemon may be stale (binary rebuilt at 2026-04-24T15:59 but daemon started at 14:41).
  Restart with: hex sched daemon restart
  Results may not reflect current code.
```

The command proceeds but the user is warned.

### Layer 3 — Startup freshness gate

When `hex sched daemon --background` starts, it records its binary mtime and PID. If the binary's current mtime differs from what's stored (from a previous run), the fresh daemon logs:

```
⬡ Starting fresh daemon (binary was rebuilt since last run).
```

This distinguishes "fresh start" from "daemon was already running" (where PID mismatch triggers a "daemon already running" error).

### Implementation details

**Staleness record location**: `~/.hex/daemon-staleness.json` (not `brain-state.json` — that file holds queue state, not process metadata).

**Binary path detection**: Use `std::env::current_exe()` to get the running binary's path. This works for both dev builds (`target/debug/hex`) and release builds (`target/release/hex`).

**Tick-skip behavior**: When stale, the daemon does NOT execute the tick body. It still sleeps and emits a warning. This prevents stale-while-running for one tick cycle. The operator must restart to restore execution. The tick skip is logged: `⬡ skipped tick (binary stale)`.

**Race condition prevention**: The mtime record is written atomically (write to temp file, rename). The startup check reads the record before writing to detect concurrent restarts.

**Daemon restart command**: `hex sched daemon restart` — stops the old daemon (via PID file), starts a new one. Implemented as:
```rust
// Read current PID from ~/.hex/brain-daemon.pid
// SIGTERM that PID
// Wait 1s for graceful shutdown
// exec `hex sched daemon --background`
```

## Consequences

**Positive:**
- Stale daemon detected within one tick cycle (≤ 30 seconds by default)
- Operator warned before acting on potentially stale results
- Daemon skips tick execution when stale — no wrong results from stale code
- Distinguishable from "daemon crashed" because PID is recorded

**Negative:**
- Adds ~1 filesystem stat call per tick (negligible overhead)
- Requires `daemon-staleness.json` to be tracked in `.gitignore`

**Mitigations:**
- The staleness check is opt-out-able via `DAEMON_STALENESS_CHECK=off` env var for CI environments where rebuilds are intentional restarts
- The tick-skip on staleness can be disabled via `DAEMON_STALENESS_SKIP_TICK=false` for operators who want continuous execution during development
