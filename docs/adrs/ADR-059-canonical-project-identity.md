# ADR-059: Canonical Project Identity Contract

**Status:** Accepted
**Date:** 2026-03-22
**Drivers:** Dashboard showed 0 swarms/agents for projects because SpacetimeDB entities stored project names while the registry stored name+suffix IDs. Every component that filters by project had a different matching strategy, all broken differently.

## Context

A "project" is referenced by at least 5 different systems:

| System | What it stores | Example |
|--------|---------------|---------|
| SpacetimeDB `project` table | `project_id` (name + random suffix) | `hex-intf-1xq8wun` |
| SpacetimeDB `swarm` table | `project_id` (often just name) | `hex-intf` |
| Agent registry | `projectDir` (filesystem path) | `/Volumes/.../hex-intf` |
| CLI `hex swarm init` | `--project` flag (name or ID) | `hex-intf` |
| Dashboard URL | hash param | `#/project/hex-intf-1xq8wun` |

When a component does `swarm.project_id === project.id`, it compares `"hex-intf"` against `"hex-intf-1xq8wun"` and gets `false`. Every project-scoped view shows "No data."

This is not a data migration problem — it's a **missing contract**. No ADR specifies what a project identity IS, how it's generated, or how entities reference it.

## Decision

### 1. Canonical Project Identity

Every project has exactly **three identifiers**, each with a specific purpose:

| Identifier | Format | Generated By | Used For |
|-----------|--------|-------------|----------|
| **`id`** | `{name}-{7char}` | SpacetimeDB `register_project` reducer | Primary key, URL params, database references |
| **`name`** | slug (lowercase, hyphens) | User or auto-derived from path | Display, human reference, CLI shorthand |
| **`path`** | absolute filesystem path | User at registration | Filesystem operations (git, analysis, scaffolding) |

**The `id` is the canonical reference.** All entities that belong to a project MUST store the full `id`, not the `name` or `path`.

### 2. Registration Contract

When `register_project` runs:
1. Generate `id` = `{name}-{random7}`
2. Store `{id, name, path}` in the `project` table
3. Return `id` to the caller

All subsequent operations (swarm_init, agent_register, task_create) MUST pass the full `id` as `project_id`.

### 3. Resolution Rules

When a system receives a project reference that is NOT a full `id`:

```
Input: "hex-intf"           → Look up by name → resolve to "hex-intf-1xq8wun"
Input: "/path/to/hex-intf"  → Look up by path → resolve to "hex-intf-1xq8wun"
Input: "hex-intf-1xq8wun"   → Exact match     → use directly
Input: ""                   → Unscoped         → does not belong to any project
```

Resolution happens **at the boundary** (CLI, MCP tool, REST endpoint), NOT inside components. By the time data reaches SpacetimeDB or the dashboard, it MUST be the canonical `id`.

### 4. Matching Utility (Frontend)

Until all producers are fixed to store canonical IDs, the frontend uses a shared matching utility (`utils/project-match.ts`):

```typescript
import { entityBelongsToProject } from '../utils/project-match';

// Use this EVERYWHERE instead of === comparison
const projectSwarms = swarms().filter(s => entityBelongsToProject(s, projectId));
```

This utility handles legacy data (name-only refs, path refs) while the system converges on canonical IDs. **No component should ever do raw `=== projectId` comparison on SpacetimeDB data.**

### 5. Enforcement

| Layer | Enforcement |
|-------|-------------|
| **SpacetimeDB reducers** | `swarm_init`, `task_create`, `agent_register` validate `project_id` exists in `project` table before accepting |
| **CLI/MCP** | Resolve name/path → canonical ID before calling reducers |
| **Dashboard** | All filtering uses `entityBelongsToProject()` — no raw `===` |
| **hex analyze** | New rule: grep for `=== projectId` or `=== props.projectId` in component filters → flag as violation |

### 6. Migration

Existing entities with empty or name-only `project_id`:
- **Do not backfill** — old swarms/agents will age out naturally via heartbeat timeout
- **New entities** created after this ADR MUST use canonical IDs
- The frontend matching utility handles the transition period

## Consequences

### Positive
- Single source of truth for what a "project identity" is
- Components can reliably filter data — no more "0 swarms" on projects with active swarms
- CLI resolution means users can type `hex-intf` and the system finds the right project
- Enforceable by SpacetimeDB reducer validation and `hex analyze`

### Negative
- Requires updating all CLI commands and MCP tools to resolve before calling reducers
- Frontend matching utility adds complexity (but it's shared, not per-component)
- Old unscoped entities won't retroactively appear on projects
