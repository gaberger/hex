# ADR-2603232220: Developer Audit Report — Full Pipeline Traceability

**Status:** Accepted
**Date:** 2026-03-23
**Drivers:** Developers need a single command to see the full lifecycle of any `hex dev` session — from the initial ask through ADR, workplan, swarm, task generation, code generation, and completion. All this data already exists in SpacetimeDB and session files but there's no unified view.

## Context

`hex dev` generates a rich data trail across multiple systems:

| Data | Source | Contains |
|------|--------|----------|
| Session | `~/.hex/sessions/dev/<id>.json` | Feature description, phase, cost, tokens, timestamps, model selections, tool calls |
| ADR | `docs/adrs/<file>.md` | Architecture decision, generated by inference |
| Workplan | `docs/workplans/<file>.json` | Steps, tiers, dependencies, adapter assignments |
| Swarm | SpacetimeDB `swarm` table | Swarm ID, name, topology, status, timestamps |
| Tasks | SpacetimeDB `swarm_task` table | Task ID, title, status, agent, result, timestamps |
| Inference | SpacetimeDB `inference_response` table | Model, tokens, cost, duration per call |
| Architecture | `hex analyze` / `GET /api/analyze` | Health score, boundary violations, dead code, cycle detection |

Today a developer has to run 5+ commands (`hex dev list`, `hex swarm status`, `hex task list`, `hex plan status`, read session JSON) to piece together what happened. A single `hex report` command should assemble this into a complete audit trail.

## Decision

Add `hex report <session-id>` that produces a complete developer audit report for any `hex dev` session.

### Data Assembly

The report command assembles data from multiple sources:

1. **Session file** (`~/.hex/sessions/dev/<id>.json`) — timestamps, cost, tokens, phase, paths, tool calls
2. **ADR file** (from `session.adr_path`) — line count, existence check
3. **Workplan file** (from `session.workplan_path`) — step count, tier breakdown
4. **HexFlo swarm** (from `session.swarm_id` → `GET /api/swarms/active`) — status, task states
5. **Generated files** (from `session.completed_steps` + workplan step files) — existence, size
6. **Architecture analysis** (`GET /api/analyze` or `hex analyze --json`) — health score, violations, dead code, cycles
7. **Tool call log** (from `session.tool_calls[]`) — per-call timestamp, phase, model, tokens, cost, duration, status

### CLI Interface

```bash
# Report for a specific session
hex report <session-id>

# Report for the most recent session
hex report --latest

# JSON output (for dashboards/CI)
hex report <session-id> --json

# List sessions to pick from
hex report --list
```

## Implementation

Primary file: `hex-cli/src/commands/report.rs`

### Architecture Analysis in Report

The report runs `hex analyze` (or calls `GET /api/analyze`) against the output directory and includes:

| Metric | Source | Description |
|--------|--------|-------------|
| Health Score | `hex analyze --json` | 0-100 composite score |
| Boundary Violations | Boundary checker | Cross-layer imports that break hex rules |
| Circular Dependencies | Cycle detector | Import cycles between modules |
| Dead Exports | Dead code analyzer | Public functions/types never imported |
| Files Analyzed | Tree-sitter parser | Count of source files parsed |
| Import Rules Checked | Boundary checker | Total import edges validated |
| Layer Compliance | Boundary checker | % of imports that follow hex rules |

The analysis runs on the **output directory** (e.g. `examples/todo-api/`) not the parent project, so it validates the generated code in isolation.

If the validate phase auto-fixed violations, the report shows both the original violations and the fixes applied, with before/after diffs.
