# weather-cli — Built Entirely by Local AI

A weather CLI app generated on a remote GPU workstation (bazzite) using hex's tiered inference routing. **No cloud API keys were used.** Every line of code was written by qwen2.5-coder:32b running on local Ollama.

## How It Was Built

This app was generated in 5 steps, each classified by task tier and routed to the appropriate local model on a Strix Halo GPU box (Vulkan, ~11 tok/s on 32B models).

### Step-by-step build log

| Step | Tier | Model | Task | Result | Tokens |
|:-----|:-----|:------|:-----|:-------|-------:|
| 1 | T2 | qwen2.5-coder:32b | Data structures (WeatherData, Forecast, Display impls) | PASS (first attempt) | ~150 |
| 2 | T2 | qwen2.5-coder:32b | CLI argument parser (parse_args, CliArgs) | PASS (first attempt) | ~130 |
| 3 | T2 | qwen2.5-coder:32b | Mock weather data provider | FAIL (mismatched brace) | ~200 |
| 4 | T2.5 | qwen2.5-coder:32b | Wire everything into main.rs | FAIL (inherited brace bug) | ~300 |
| 5 | — | qwen2.5-coder:32b | Error-feedback retry with compile error | PASS | ~200 |

**Total: 2 clean passes + 1 retry = working app.** The mismatched brace in step 3 was caught by the `rustc` compile gate and fixed automatically via error-feedback retry — the compiler error was fed back to the model, which corrected it on the next attempt.

### What this demonstrates

1. **Tiered routing works for real apps.** Each module was a T2 task (single function/struct generation). The wiring step was T2.5 (multi-module composition). A T3 frontier model was never needed.

2. **Compile gates catch real bugs.** The model generated a mismatched parenthesis: `((hash % 5) as f64 - 2.5` — missing the closing `)`. The `rustc` gate caught it, and the error-feedback retry fixed it on the next pass.

3. **Error-feedback retry is the safety net.** When Best-of-N fails, the model gets the compiler error and corrects itself. Research shows 15-25% improvement at 32B scale, and we observed it here: 0% → 100% on the retry.

4. **Remote execution is practical.** The entire build ran on bazzite over SSH. The Mac coordinator never touched the code — it was generated, compiled, and tested entirely on the remote GPU box.

### Build environment

- **Machine**: Strix Halo (AMD Ryzen AI, Vulkan GPU)
- **OS**: Bazzite (Fedora-based Linux)
- **Model**: qwen2.5-coder:32b (Q4_K_M, 19GB)
- **Inference**: Ollama 0.20.2, ~11 tok/s
- **Compiler**: rustc 1.86 (edition 2021)
- **Cloud APIs used**: None
- **Cost**: $0.00

## Usage

```bash
# Build
rustc --edition 2021 main.rs -o weather-cli

# Current weather
./weather-cli London
# → Weather in London: -1.5°C, Humidity: 76%, Sunny, Wind: 40.0 km/h

# 5-day forecast
./weather-cli Tokyo --forecast
# → Day 1: High 7.5°C, Low -7.5°C, Cloudy
# → Day 2: High 8.5°C, Low -6.5°C, Rainy
# → ...

# Fahrenheit
./weather-cli NYC -u f

# Usage help
./weather-cli
# → Usage: weather_cli <city> [--forecast] [-u f|c]
```

## Architecture

```
main.rs (146 lines, single file)
├── WeatherData + Display    — generated by T2 task
├── Forecast + Display       — generated by T2 task  
├── CliArgs + parse_args()   — generated by T2 task
├── get_weather()            — generated by T2 task
├── get_forecast()           — generated by T2 task
└── main()                   — generated by T2.5 task (wiring)
```

All code lives in a single file for simplicity. A real hex workplan would decompose this into separate modules in isolated git worktrees (domain → ports → adapters), but the single-file approach demonstrates the inference pipeline without the orchestration overhead.
